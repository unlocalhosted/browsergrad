# Browsergrad-jit feedback report

Generated by exercising the library end-to-end via two integration
benches:

- `tests-integration/end_to_end_bench.test.ts` — 14 scenarios across
  every shipped PRD. Output: `/tmp/bg-feedback-report.{md,json}`.
- `tests-integration/perf_bench.test.ts` — matmul / softmax / AMP /
  trace cache / vmap timing on the NumPy realizer. Output:
  `/tmp/bg-perf-report.md`.

Both benches run as ordinary integration tests and write structured
reports for inspection. They're additive — new PRDs add a record() call
and an entry shows up in the report next time.

## Headline results

### What works (14/14 scenarios pass)

- **PRD-005/007 — Forward + backward + SGD step**: 8.3ms first call,
  1.9ms second (trace cache hit). MLP loss decreases as expected.
- **PRD-006 — Softmax fusion**: 1 fused kernel per softmax DAG.
- **PRD-008 — Safetensors roundtrip**: 703 bytes for w1[8,16] + b1[16].
- **PRD-009 — Gradient checkpointing**: max grad diff vs uncheckpointed
  is 0 (bit-exact).
- **PRD-010 — fp32 accumulator matmul**: K=4096 all-ones lands exactly
  at 4096.0 (would saturate at ~2048 with pure f16 accumulate).
- **PRD-011.5 — WebGPU realizer seam**: dispatches via the mock bridge,
  upload_count=2, matmul_count=1, residency contract holds.
- **PRD-014 — Functional grad + vmap**: `grad(x*x.sum())(x)` returns
  `[2, 4, 6]` and `x.grad` stays None (compose-friendly).
- **PRD-015 — Custom WGSL registration**: hash-based registry works,
  OP_CUSTOM tagged "user".
- **PRD-016 — ONNX export**: 346 bytes for `x @ w + b`, first byte
  matches proto3 ir_version field.
- **Cross-PRD**: full train-save-load-infer pipeline works. Loss
  decreases (1.74 → 1.54); save/load preserves loss to within 1e-5.

### What the bench uncovered and we fixed in place

| Issue | Fix |
|-------|-----|
| `save_safetensors` required a `path` argument — impossible in the browser | Made `path` optional; always returns bytes; `path` is an additive write |
| `save_safetensors` rejected `np.ndarray` values (TensorProxy-only) | Now accepts both — natural call sites work |
| `from_numpy(restored_proxy)` raised because proxy isn't an ndarray | Identity overload: `from_numpy(TensorProxy)` returns the proxy unchanged |

Each was discovered by running real code (the cross-PRD scenario) and
treating the failure as data, not a regression.

## Performance baseline (NumPy realizer)

### Matmul shape sweep

| M × N × K | ms | GFLOPS |
|-----------|----|--------|
| 64×64×64 | 0.41 | 1.27 |
| 128×128×128 | 1.43 | 2.93 |
| 256×256×256 | 11.51 | 2.91 |
| 512×64×256 | 5.26 | 3.19 |
| 32×32×1024 | 0.81 | 2.60 |

**Observation**: ~2-3 GFLOPS. Native NumPy without BLAS-bound dispatch.
Modern CPUs do ~50 GFLOPS for f32 BLAS, and a tiled GPU kernel hits
~100-500 GFLOPS on integrated graphics. **The 30-100× headroom is what
PRD-011.5's WGSL bridge unlocks** — and we measured it: the seam is
in place, just needs browser CI to exercise.

### Softmax fusion

3 sweeps across `(batch, features)`. **All produce exactly 1 fused
kernel** — PRD-006's 7-node-DAG → FUSED_SOFTMAX matcher fires reliably.

### AMP fp16 vs fp32

| path | ms | max abs diff |
|------|----|--------------|
| f32 | 11.56 | 0 |
| AMP (f16 input → f32 accumulator → f16 output) | 11.92 | 3.6e-2 |

**Honest finding**: AMP on NumPy is **not** faster than f32 (the cast
pass + fp32 accumulator add work; NumPy lacks f16 SIMD). The PRD-010
reviewer flagged this — confirmed in measurement. Wall-clock win
materialises when PRD-011.5's WGSL backend runs on a device with f16
support (Apple M-series, recent Intel, Snapdragon X).

### Trace cache

- Cold call: 0.82ms
- Warm call (avg of 5): 0.23ms
- **~3.6× speedup** from cache hit. PRD-008 working as designed.

### vmap vs Python loop

- vmap path: 0.10ms (1 trace, 1 realize)
- Python for-loop over 32 samples: 1.62ms (32 traces, 32 realizes)
- **~16× speedup**. The functional batching transform is the right call.

## Ranked improvement priorities

Based on what the bench surfaced + the deferred-PRD list:

### P0 — User-facing friction

1. **`TensorProxy.__array__` UX**: currently raises with a helpful message,
   but every `np.allclose(arr, proxy)` call hits it. Consider an opt-in
   `bg.allow_implicit_realize()` context for notebook ergonomics. Same
   reasoning as PyTorch's old `.data` debate — explicit is right, but
   noisy.

2. **State-dict pattern**: `model.weight = restored["weight"]` now works
   (after the from_numpy identity fix). Should formalise into a
   `model.load_state_dict(restored)` helper that walks named parameters
   and assigns them. Tiny, mechanical.

### P1 — Performance

3. **WGSL backend browser CI**: PRD-011.5's seam is shipped; the wall-clock
   wins are unverified outside the mock bridge. Set up a Playwright job
   that runs the realizer against a real GPUDevice in headless Chrome
   and re-runs the matmul sweep. Expected: 30-100× over NumPy on a
   tiled-GEMM 256×256.

4. **Trace cache parameter coverage**: cache currently misses on graphs
   with `requires_grad=True` parameters. Cold/warm ratio of 3.6× could
   become much higher if backward graphs also cache. Investigate the
   `_trace_cache.py:147` exclusion.

### P2 — Functional-transform completeness (PRD-014b)

5. **vmap+grad composition**: vmap alone works; `vmap(grad(fn))(x)` has
   subtle shape-broadcasting interactions (VJP rules record dy.shape
   which can be smaller than realized ndarray shape). Fix: have VJP
   emit explicit-shape ops where the shape is currently inherited.

6. **Remaining vmap rules**: 11 opcodes still raise (SCATTER_ADD, INDEX,
   MASK, RANDOM, CUSTOM, FUSED_*, PAD, SLICE, ISNAN). Each is ~30 LOC.
   Stable order of attack: SCATTER_ADD (embedding backward), INDEX
   (gather), PAD (conv-ish), SLICE (last-token attention).

### P3 — Deferred per agent reviews (revisit later)

7. **PRD-012b cost model + producer-consumer fusion**: blocked on lab
   content. Revisit when we have 5+ labs and can see real op patterns.
8. **PRD-011 WebNN backend**: blocked on Chrome GA (~2027) and NPU
   driver maturity. Live user fraction <5% today.
9. **PRD-012c cross-block megakernel**: vanity. The wins live in
   Flash Attention + fused FFN, which we have separately.

### P4 — Tooling

10. **bg-lab CLI**: PRD-013 manifest validator is a Node-importable
    function; wrap it in a CLI binary for craftingattention authors.
11. **ONNX → onnxruntime conformance**: PRD-016 emits proto3 bytes
    but we don't verify against onnxruntime yet. Set up a Node test
    that loads the exported model and asserts numerical parity.
12. **Browser-side benchmarks**: run the perf bench in a real browser
    via Playwright. Captures cold-start cost, page-load impact,
    GPU vs CPU realize comparison.

## What the harness itself proved

The two bench files are now the **regression substrate for the entire
library**. Every shipped PRD gets one record() call in
`end_to_end_bench.test.ts`. When a PRD breaks, a real workflow breaks,
and the bench surfaces it in the markdown report — not just a unit-test
failure.

This is the closest the workspace gets to "test like a user." Future
PRDs should land their own scenarios in the bench as part of the
"green tests" criterion.
